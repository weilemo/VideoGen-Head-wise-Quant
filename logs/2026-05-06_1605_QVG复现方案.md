# QVG 复现方案

日期：`2026-05-06`

## 目标

- 复现 `Quant VideoGen (QVG)` 的核心结论：在长视频自回归生成中，用 `2-bit KV-cache quantization` 将 `KV cache` 显存压缩到约 `1/7`，同时尽量保持视频质量，仅带来很小的端到端额外开销。

## 本次已确认的信息

- 论文：`Quant VideoGen: Auto-Regressive Long Video Generation via 2-Bit KV-Cache Quantization`
- arXiv 页面显示当前版本为 `v4`，更新时间为 `2026-05-01`
- 官方仓库就在当前工作区：`workspace/videoquant/Quant-VideoGen`
- 官方 README 明确给出了三条实验集成：
  - `LongCat-Video`
  - `Self-Forcing`
  - `HY-WorldPlay`
- 官方 README 还给出了对应的 `BF16` 与 `QVG INT2` KV-cache memory 结果：
  - `LongCat-Video`: `22272.00 MB -> 3231.28 MB`，约 `6.89x`
  - `Self-Forcing`: `46072.88 MB -> 6613.59 MB`，约 `6.97x`
  - `HY-WorldPlay`: `29700.00 MB -> 4235.45 MB`，约 `7.01x`

## 论文实验的复现范围

基于论文摘要、官方 README、官方脚本入口和代码结构，当前建议把复现分成三层。

### 第一层：核心结果复现

- 目标：先复现论文最核心、最确定的结果，也就是 `BF16` 对 `QVG INT2` 的质量和显存对比。
- 覆盖模型：
  - `LongCat-Video`
  - `Self-Forcing`
  - `HY-WorldPlay`
- 核心输出：
  - 生成视频样例
  - 每层 `KV cache` 显存
  - 总 `KV cache` 显存
  - 端到端运行时
  - 定性质量对比，重点看 `identity / scene / motion`

### 第二层：方法消融复现

- 目标：验证 `QVG` 的关键设计是否真的有效。
- 建议优先做的消融：
  - `BF16`
  - `naive-int2`
  - `kmeans-int2`
  - `triton-nstages-kmeans-int2`
  - `triton-nstages-kmeans-int2-clip`
- 重点回答：
  - 仅做低比特量化是否就会明显伤害质量
  - `PRQ` 是否比单阶段量化更稳
  - clipping 版本是否能缓解极端值导致的质量退化

### 第三层：参数敏感性复现

- 目标：确定论文默认配置附近的稳定区间。
- 建议扫描：
  - `num_prq_stages`: `1 / 2 / 4`
  - `quant_block_size`: `32 / 64 / 128`
  - `cache_num_k_centroids`: `128 / 256 / 512`
  - `cache_num_v_centroids`: `128 / 256 / 512`
  - `quant_type`: `int2 / int4`
- 重点观察：
  - 显存压缩比
  - 生成耗时
  - `motion continuity`
  - `subject consistency`

## 代码层已确认的方法对应关系

- `Semantic-Aware Smoothing` 在当前仓库中主要体现在：
  - `quant_videogen/real/smooth.py`
  - 通过先聚类、再减去对应 centroid，生成更适合量化的 residual
- `Progressive Residual Quantization (PRQ)` 在当前仓库中主要体现在：
  - `quant_videogen/real/prq.py`
  - `quant_videogen/functions.py`
  - 多阶段 `k-means` 后再对最终 residual 做低比特量化
- `KV cache` 的量化存储与按需解压在：
  - `quant_videogen/kv_cache.py`
